## Definition of Done (v1)

- API /health returns 200 (DB reachable; in local dev, Kinesis check is skipped)
- End-to-end event flow: `ingest.py` → DMS → Kinesis → Matching Engine Lambda → SQS → Dispatcher Lambda → Kapso
- Correlation ID (UUID) propagated through entire pipeline (API → Kinesis → SQS → Dispatcher)
- All unit tests green, integration test against LocalStack passing
- SQS DLQ configured: maxReceiveCount=3, backoff delays 2s/4s/8s, CloudWatch alarm on DLQ depth > 0
- CDK `cdk deploy` provisions all automatable infrastructure (see CDK scope below)
- DMS configured manually per runbook (separate from CDK)
- Matching query returns in <50ms at 100k alert rules (verified via EXPLAIN ANALYZE)
- Pre-deploy checklist: Lambda concurrency quota, Kinesis shard quota confirmed

## Constraints

- **Immutable pipeline:** No changes to ingestion code or `weather_data` table schema. RDS parameter group changes (enabling `rds.logical_replication = 1` for DMS WAL reading) are DB admin operations, not pipeline modifications.
- **Lambda concurrency:** Matching Engine Lambda reserved concurrency capped to prevent saturating RDS Proxy. Start with 10 concurrent executions, 2 Kinesis shards. Scale formula: 1 shard = 1 concurrent consumer. Revisit at > 500 events/sec sustained.
- **H3 resolution:** Fixed at resolution 7 (~5.16 km2 per cell). Balances matching precision with index cardinality. Stored as constant `H3_RESOLUTION = 7`.
- **Alert cooldown:** `ALERT_COOLDOWN_MINUTES = 15`. Per-rule cooldown via `last_notified_at` column. Dispatcher skips notification if `now() - last_notified_at < 15 min`.

## Schema Contracts to Lock

- `weather_data` (read-only contract from black-box pipeline): **Spike task required** before implementation. Introspect live table to lock exact column names, types, and whether location is float pair or PostGIS geometry. Assumed minimum: `id`, `location_lat` (float), `location_lon` (float), `metric_type` (varchar), `value` (numeric), `recorded_at` (timestamp with tz).
- `AlertRules`: `id` (bigint PK), `user_id` (bigint FK), `location_h3_index` (varchar, H3 res 7), `metric_type` (varchar), `operator` (varchar, one of: gt, gte, lt, lte, eq), `threshold_value` (numeric), `last_notified_at` (timestamp with tz, nullable), `created_at` (timestamp with tz), `updated_at` (timestamp with tz)
- Composite index on `AlertRules(location_h3_index, metric_type)` for <50ms matching queries (threshold_value excluded, fetch-and-filter evaluates in Python)

## Kapso API Contract

- Endpoint, auth mechanism, and payload schema to be documented during implementation spike

## Vision (OUT OF SCOPE FOR v1, future roadmap only)

### 10x Check
The 10x version of Agrobot isn't just threshold alerts, it's a weather intelligence platform. Users don't just get "temp > 35" notifications. They get predictive alerts ("heat wave approaching your region in 6 hours"), correlated multi-metric alerts ("high temp + low humidity = fire risk"), and historical pattern analysis ("this weather pattern historically damages wheat crops"). The matching engine becomes a rules engine that supports compound conditions, temporal patterns, and ML-derived thresholds.

## Accepted Scope (added to this plan)

- **Pluggable NotificationProvider interface with KapsoProvider**
  Interface: `class NotificationProvider: async def send(self, user_id, rule_id, triggered_value, correlation_id) -> bool`
  The `send()` method receives post-validation enriched data (after Claim Check against RDS replica), not the raw SQS message. Flow: SQS message → Dispatcher validates against RDS replica → calls `provider.send()` with enriched payload.
- **GET /health and GET /status endpoints**
  /health: DB reachable (production adds Kinesis stream check; local dev skips it via env flag). /status: iterator age, SQS depth, DLQ count, last dispatch timestamp.
- **AWS CDK infrastructure**
  Stacks: VPC, RDS (+ parameter group for logical replication), Kinesis (2 shards), SQS (+ DLQ with maxReceiveCount=3), Lambda (Matching Engine + Dispatcher, with `bisectBatchOnFunctionError: true` and `batchItemFailures` response), ECS Fargate service, API Gateway, IAM roles. **Excluded from CDK:** DMS (configured manually per runbook), ElastiCache (not needed until Redis cooldown).
- **Realistic weather data simulator with scenario modes**
  Named scenarios: `HEAT_WAVE` (temp > 38C, humidity < 20%), `COLD_SNAP` (temp < -5C, wind > 60km/h), `SEVERE_STORM` (burst of 10k+ events across H3 k-ring radius 2 from Buenos Aires center in 30 seconds). Uses real coordinates (Buenos Aires, Sao Paulo, Mexico City).
- **JWT authentication (FastAPI-issued) + rate limiting**
  JWT tokens issued by FastAPI /auth/login endpoint. Signing secret in AWS Secrets Manager. Rate limit enforced at both API Gateway (Usage Plans) and app layer (FastAPI middleware as fallback). /status endpoint admin-scoped (is_admin JWT claim).

## Error Handling: Matching Engine Lambda

- **Partial batch failures:** Use aws-lambda-powertools Batch Processor (not manual batchItemFailures). `bisectBatchOnFunctionError: true` in event source mapping.
- **RDS query timeout:** Log with correlation_id, return record as failed item (will be retried via Kinesis). After 3 retries, record ages out per Kinesis retention.
- **No rules match:** Normal case. Log at DEBUG level, acknowledge record. No SQS publish.
- **SQS publish failure:** Return record as failed item for retry. Log at ERROR with correlation_id.
- **Unknown operator in DB:** Skip rule, log ERROR. Prevents unhandled exceptions from direct DB inserts or migrations.

## Eng Review Findings (2026-03-31)

### Architecture Changes
1. **Matching Engine reads from Read Replica** (not primary). Primary reserved for API writes + Dispatcher cooldown writes.
2. **Two RDS Proxy endpoints** in CDK (primary + replica).
3. **Dispatcher uses two DB engines:** REPLICA_DATABASE_URL (Claim Check reads) + PRIMARY_DATABASE_URL (cooldown writes).

### Implementation Details
4. Use aws-lambda-powertools Batch Processor for Kinesis partial batch failures.
5. `pool_pre_ping=True` on all SQLAlchemy engines (stale connection handling after failover).
6. h3 C extension built via Docker bundling in CDK PythonFunction (--platform linux/amd64).
7. CDK assertions tests alongside snapshot tests (Lambda env vars, SQS DLQ config, etc.)
8. correlation_id in SQS message body. Standard queue (not FIFO).
9. Runbook entry for bisection throttling scenario.
10. **Prerequisite #1 expanded:** weather_data schema spike now includes DMS Kinesis event envelope format discovery.
11. Dispatcher SQS batch size = 10 with asyncio.Semaphore for concurrent Kapso calls. Tune after Kapso spike.
12. `operator` validated as Python Enum in Pydantic; Matching Engine also validates unknown operator from DB.

### Index Correction
- Composite index changed to `(location_h3_index, metric_type)`. `threshold_value` removed (fetch-and-filter never queries on it in SQL).
